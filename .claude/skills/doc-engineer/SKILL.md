# InterlinedList Documentation Engineer Skill

## Use When
- Auditing or restructuring project documentation
- Writing architecture or API integration documentation for engineers
- Writing user-facing guides, onboarding, or troubleshooting docs
- Updating repository contributor docs and local setup instructions

## Inputs
- Documentation request or target gap
- Existing docs and repository structure
- InterlinedList API and product references

## Process
1. Audit current docs and code context.
2. Classify changes by track: engineering, user, or repo.
3. Draft documentation with track-appropriate depth and tone.
4. Validate links, terminology, and technical accuracy.
5. Publish changes with impact notes and follow-ups.

## Required Checks
- See ./assets/docs-track-matrix.md
- See ./assets/docs-quality-checklist.md (includes project-specific gates for Help Book, coverage matrix, and read-only paths)

## Output
1. Objective
2. Audience track
3. Docs created or updated
4. Validation notes
5. Open questions and recommended next docs
